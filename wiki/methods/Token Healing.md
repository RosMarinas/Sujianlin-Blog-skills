# Token Healing

## 适用问题

在基于大词表（极大化压缩率）的大语言模型（LLM）上进行代码补全或文本续写时，由于用户的输入（Prompt）在任意位置被截断，原本应该被分词器（Tokenizer）作为一个整体长 Token 处理的常见短语，会被强制切割成短的子词，导致模型在续写时严重偏离预期（例如遇到 `import numpy` 无法补全 ` as np`，遇到 `白云` 无法补全 `机场`），这种现象在代码模型或重度合并多字词的模型中尤为显著。

## 核心变换

将“基于完整输入的分词进行自回归解码”转化为“回退到上一个边界，利用词表约束下的条件概率重新选择最优词组”。即不是盲目相信对被截断输入的分词结果，而是通过“回退一步 + 前缀搜索”，把模型对未来单个字的盲目预测，转换成了对包含该前缀的所有长 Token 的概率打分和重新采样。

## 典型步骤

1. **常规分词与剥离**：对用户给定的未完成输入文本进行分词。然后从序列末尾丢弃最后一个 Token（如果截断可能跨越多个短 Token，甚至需回退多步），记录其对应的原始字符串前缀。
2. **构建候选集**：在 Tokenizer 词表中进行简单的字符串前缀匹配搜索，找到所有以此字符串开头的 Token（这部分在 CPU 上纯查表完成，代价极低）。
3. **计算条件概率**：将丢弃末尾 Token 后的前序上下文送入 LLM 进行前向计算，得到下一个位置关于全词表的概率分布。
4. **截断归一化**：仅提取上一步在词表里搜出的那些候选 Token 的概率值，重新进行 Softmax 归一化。
5. **采样与修补**：从归一化后的候选集中采样出一个目标 Token。由于采样出的长 Token 必然包含原被截断的输入字符串作为前缀，直接将该长 Token 对应的字符串作为续写输出（修补补全），随后再恢复正常的自回归解码。

## 直觉

Tokenizer 在面对未完成的句子时是“短视”的，它不知道后面还有什么内容，只能把眼前的字符切碎。而大语言模型在训练时见过完整的“import numpy as np”作为一个长 Token 出现。如果不做 Token Healing，我们相当于强迫模型接上一个它平时根本不会用的碎 Token；如果做 Token Healing，我们就是帮 Tokenizer 开了一个“透视挂”：先问问 LLM“你觉得基于前文，这个只写了一半的词，最可能是词表里哪个完整的大词组？”一旦 LLM 选出来，我们就直接把这个大词组补上去。不仅修复了分词错误，还因为大词组自带更多后缀，使得模型一次性生成了更多的字，把大词表容易因截断造成的缺点直接变成了生成效率提升的优点。

## 边界

- **回退步数选择**：通常回退一个 Token 足以解决大部分问题，但如果是极端的分词器（比如 BPE 导致前缀切分极为细碎），可能需要回退 2-3 个 Token 才能找到真正期望的词表候选项，这需要根据具体的 Tokenizer 特性调试。
- **纯粹的新词补全**：若用户输入的截断前缀本身就是一个生僻词或 OOV（未登录词），词表前缀搜索可能只返回原有的零碎 Token 甚至是空集，此时 Token Healing 会退化为普通的解码，但在大词表场景下这种情况较少影响其正面作用。

## 例子

假设用户的代码输入停留在 `import numpy`。由于代码大词表模型通常将 `import numpy as np` 整体编为一个 Token，单纯输入 `import numpy` 可能会被切分成 `import / numpy`。接下来正常续写时，由于模型在预训练时从没见过单独的 `numpy` 后面接 ` as np`（因为都被合并了），导致它无法正确补全。
使用 Token Healing：
1. 回退一步，剥离 `numpy`，上文变为 `import`。
2. 拿 `numpy` 去词表做前缀搜索，找到候选集：`numpy`、`numpy as np`、`numpy.random` 等。
3. 利用 LLM 给定上下文 `import`，计算这几个候选词的概率。由于预训练数据中 `import numpy as np` 极高频，它的概率极高。
4. 采样选中 `numpy as np`。
5. 输出 ` as np` 完成修补，并继续预测。
